1. Overview

Sistema di monitoraggio del traffico in tempo reale che rileva e conta 7 categorie di veicoli/pedoni usando un modello YOLOv8n custom addestrato su VisDrone2019-DET. Include una pipeline live (webcam / video file) con una linea di conteggio virtuale.

🔧 Tech Stack

  • Python 3.10+
  • OpenCV — cattura video, visualizzazione
  • Ultralytics YOLOv8 — object detection
  • PyTorch / TorchVision — backend deep learning
  • pytest — suite di test

📁 File inclusi out-of-the-box

  • models/best.pt — pesi custom VisDrone (7 classi)
  • models/yolov8n.pt — pesi COCO fallback
  • demo/traffic_demo.mp4 — video di test aereo

5.3 src/webcam/pipeline.py

Orchestra l'intera pipeline frame-by-frame.

  1. Acquisisce frame dallo streamer
  2. Esegue detection con YOLO
  3. Aggiorna il tracker con le detection
  4. Verifica i crossing sulla linea
  5. Accumula i conteggi
  6. Disegna bounding box, linea e contatori
  7. Sovrappone l'HUD con gli FPS

5.4 src/deep_learning/yolo_detector.py

Wrapper Ultralytics YOLOv8.

    2. models/best.pt (custom)
    3. runs/detect/.../best.pt (ultimo training)
    4. models/yolov8n.pt (COCO bundled)
    5. Download automatico YOLOv8n

5.5 src/deep_learning/train_yolo.py

Funzione train_yolo(data_yaml, output_dir, epochs) che avvia il fine-tuning di YOLOv8n sui dati VisDrone.

  • Auto-detect del device: MPS (Apple Silicon) → CUDA → CPU
  • Parametri letti da src.config.YOLO

5.6 src/postprocessing/tracker.py

Il cuore del tracking e del conteggio.

Matching a due stadi

  1. Stage 1 — IoU: ogni detection viene matchata con il track esistente che ha IoU massima (≥ 0.3).
  2. Stage 2 — Centroid distance: le detection non matchate cercano un track entro 50 px di distanza euclidea (fallback per quando i bounding box cambiano forma ma il centro è stabile).
  3. Le detection ancora senza match generano un nuovo track.

Crossing bidirezionale

Il conteggio avviene solo quando un oggetto attraversa effettivamente la linea:

crossing = (prev_y < line_pos ≤ curr_y) OR (prev_y ≥ line_pos > curr_y)

Funziona sia in orizzontale che in verticale.

Persistenza

Un track rimane in memoria per 60 frame (~2 secondi a 30 fps) anche senza detection. Questo riduce drasticamente il respawning di ID quando il modello perde un oggetto per pochi frame.

5.7 src/utils/visualization.py

6. Teoria

6.1 YOLOv8 — Object Detection

YOLO (You Only Look Once) è un single-stage detector: in un'unica forward pass della rete neurale predice contemporaneamente bounding box e classi per tutti gli oggetti nell'immagine.

Architettura YOLOv8n (nano)

  • Backbone (CSPDarknet): estrae feature a diverse scale dall'immagine
  • Neck (PAN-FPN): fonde le feature a diverse risoluzioni per rilevare oggetti piccoli e grandi
  • Head: predice bounding box, confidenza e classi
flowchart LR IMG[640×640 Image] BB[Backbone] NK[Neck PAN-FPN] HD[Detection Head] OUT[Boxes + Classes + Confidences] IMG --> BB --> NK --> HD --> OUT

Anchor-free

A differenza delle versioni precedenti, YOLOv8 è anchor-free: non usa anchor box predefinite. Predice direttamente il centro, larghezza e altezza di ogni oggetto. Questo semplifica il training e migliora la generalizzazione.

6.2 Intersection over Union (IoU)

Metrica per misurare la sovrapposizione tra due bounding box:

IoU(A,B) = |A ∩ B| / |A ∪ B|

Nel tracker, l'IoU è usata per associare una detection del frame corrente al track del frame precedente. Se due box si sovrappongono per almeno il 30% (soglia configurabile), sono considerati lo stesso oggetto.

flowchart LR A[Box Frame t-1] B[Box Frame t] I[Intersezione] U[Unione] R["IoU = I / U"] A --> I B --> I A --> U B --> U I --> R U --> R

6.3 Non-Maximum Suppression (NMS)

Il modello YOLO può generare molteplici detection sovrapposte per lo stesso oggetto. NMS elimina le detection ridondanti mantenendo solo quella con confidenza più alta:

  1. Ordina le detection per confidenza decrescente
  2. Prendi la detection con confidenza più alta e conservala
  3. Scarta tutte le detection con IoU > soglia (es. 0.4) rispetto a quella conservata
  4. Ripeti fino a esaurimento

6.4 Tracking — Associazione Frame-to-Frame

Il problema del tracking online si riduce a un problema di assegnazione bipartita:

  • Insieme A: track esistenti (dal frame precedente)
  • Insieme B: detection nuove (dal frame corrente)
  • Obiettivo: trovare la corrispondenza A↔B che massimizzi l'accuratezza

Il nostro tracker usa un approccio greedy a due stadi:

  1. Prima matcha per IoU (forma precisa del box)
  2. Se l'IoU fallisce (es. box cambia forma per occlusione o movimento rapido), fa fallback per distanza dei centroidi (50 px)

Questo è più robusto di un semplice IoU tracker, specialmente in scenari aerei dove i veicoli si muovono velocemente e le prospettive cambiano.

6.5 Counting Line — Rilevamento dell'Attraversamento

Per contare solo oggetti che passano effettivamente la linea (e non quelli che nascono già oltre), il sistema memorizza il prev_centroid di ogni track:

crossing = (prev_y < line_pos ≤ curr_y) ∨ (prev_y ≥ line_pos > curr_y)

Se prev_centroid è None (oggetto appena nato), il crossing non viene contato. Questo previene conteggi falsi su detection che appaiono già oltre la linea.

7. Configurazione

Tutti i parametri sono in src/config.py. Modifica questo file per adattare il comportamento senza toccare il codice sorgente.

Linea di conteggio

WEBCAM = {
    "count_line_orientation": "horizontal",  # "horizontal" o "vertical"
    "count_line_y_ratio": 0.5,               # 0.0 = top, 1.0 = bottom
}

Soglie detection

YOLO = {
    "conf_threshold": 0.5,   # min confidenza (0.0–1.0)
    "nms_threshold": 0.4,    # soglia NMS IoU
}

Colori e stile

VISUALIZATION = {
    "box_colors": {
        0: (128, 128, 128),   # pedestrian
        1: (255, 128, 0),     # bicycle
        2: (0, 255, 0),       # car
        ...
    },
    "count_line_color": (0, 165, 255),
    "font_scale": 0.5,
    "thickness": 2,
}
